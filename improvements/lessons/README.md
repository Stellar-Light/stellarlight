# lessons/ — the mistake corpus

Every real defect we've shipped, filed by **class**, with the mechanized guard that now catches it. The point is not the history — it's that each class is a **design checklist item**: when building any new endpoint, field, or surface, walk this table and ask "does the new thing reintroduce one of these?"

Sources: sls board (`kalepail/stellar-raven/improvements/stellar-light-scout`, 18 items), our tracker issues, cold audits, user demo feedback, and our own gists/postmortems. New incidents get a row here (and a detail file when the story carries design weight) as part of closing them — same loop as filing the verification comment.

| 4 | **Order-implies-rank** — consumers read array order as meaning | sls-001/005 (winners unordered) | self-describing arrays (`placementRank`, sorted + documented) |

| 16 | **Cross-chain headline ≠ chain footprint** — an aggregator's chain membership + headline metric says nothing about the per-chain slice | [2026-07-16](./2026-07-16-slice-truth-floors-and-windows.md): NEAR Intents "$92M gap" was $11.5k on Stellar; enrich-tvl was one run from writing Sentora's $2.06B cross-chain total as Stellar TVL | `stellarTvlOf()` (chainTvls.Stellar) in enrich-tvl + the coverage report floors/ranks on the slice and prints both columns |
| 17 | **Ungated relevance floors** — a ranking floor without a discrimination condition + intent exclusion over-promotes generic coverage | [2026-07-16](./2026-07-16-slice-truth-floors-and-windows.md): full-lexical floor regressed golden 46→43 ("SCF handbook link" served 7 pages containing the words; undated version pages beat dated Protocol-27 posts) | discriminating gate (≤5 covered chunks) + recencyIntent exclusion + golden currency/known-item cases; rule: every new floor ships with both gates and a golden case |
| 18 | **Windowed guards under-report population defects** — a check that reads a result window hides failures behind the ones it names | [2026-07-16](./2026-07-16-slice-truth-floors-and-windows.md): fixing the audit's 4 named bridge rows surfaced a 5th (wowmax) the q=bridge window had hidden | when a windowed check reds, sweep the FULL population (`?type=X&limit=100`) and fix all rows in one pass |
| 19 | **URL-tier "not served" verdicts** — judging coverage by result URL alone false-negatives content served under another host | [2026-07-16](./2026-07-16-slice-truth-floors-and-windows.md): the partner-docs lane declared Alchemy "NOT SERVED" while the content was retrievable from a stellar.org-hosted chunk | tiered own/mention/none verdicts matching URL AND served text; verbatim-quote probe at high limit before concluding content is absent |
| 16 | **Adversarial evals manufacture problems** — "find the worst" judges over-rotate fixes | the 6-deploy repo-search over-rotation | grade against ground-truth answer keys (depth-labels, recall answer key, golden eval) — never adversarial judging |

| 19 | **Stale-but-close outranks current** — retrieval order ignores the freshness/authority already priced into confidence; duplicate chunks crowd the top-K | [2026-07-09](./2026-07-09-research-rank-starbridge.md): Starbridge (2022, never shipped) ×2 in the top-5 for "bridge assets from EVM to Stellar" while CCTP/Allbridge sat at ranks 11–20 | `research-rank.ts` (best-chunk-per-doc + confidence ordering, both retrieval paths); golden `forbiddenRegex: starbridge`; offline re-rank simulation on prod pools before shipping ranking changes |

| 21 | **Name-match identity** — same name ≠ same entity; audits/announcements attributed by name can belong to someone else | [2026-07-09](./2026-07-09-curation-writes-and-identity.md): the Certora "Spectra Bridge" audit — spectra.finance's site has zero Stellar mentions; identity proven only via the audit PDF's scope links → perspectivefi | verify identity via the artifact's own links (repos/domains/orgs), never name match; slug-join domain cross-check still open (review #21) |
| 22 | **Instance-calibrated fixes** — the fix matches the triggering example, not the class it belongs to | [2026-07-09](./2026-07-09-curation-writes-and-identity.md): Beacon-Q3 shipped EVM-only vocabulary + `[stellar, evm]` seeds; Solana/Tron/XRPL corridors stayed broken, Rozo invisible | enumerate sibling instances before closing (loop step 4 applied during the fix); the primary-source corridor matrix + per-class daily guards |
| 23 | **Meta-only truth (omission-by-placement)** — the answer-changing fact lives in `meta` while the rows imply the opposite; row-reading consumers conclude wrong | 2026-07-09 (Emir/SDF, live Raven session): every getRfps row read `closed` while "SCF #45 OPEN until 2026-08-02" sat only in meta.scfRound → "nothing is open"; same shape on partners (counts.total=0 hid 5 wallets behind the quality bar with no disclosure) | answer-changing facts serve as ROWS or per-row fields (open scf-round row; partners `counts.filteredOut`); self-audit asserts the meta⇔rows contract both directions; design rule: agents read rows — meta is commentary |
| 24 | **Confident fallbacks** — a lower-trust retrieval rung (vector fallback, relaxed tier) inherits the labels/confidence of the strict path, turning "no answer" into a confident wrong answer | 2026-07-10 (audit R1, #437): pure semantic-fallback pages claimed matchMode `strict` ("all keywords matched") with the top guess at 0.9+ `high` — the cosine was normalized against the fallback set's OWN max, so the best guess always read relevance≈1.0 (q=fonbank served veur/boss-pay as authoritative) | `matchMode: semantic` + honest label + `counts.semantic`; `semanticProjectConfidence` = absolute cosine band, hard-capped below `high`; daily self-audit probes assert both; design rule: every retrieval rung labels itself and caps its trust below the rung above it |
| 25 | **Hash-gated upserts strand metadata fixes** — an ingest that skips on unchanged content-hash never applies extraction fixes to fields OUTSIDE the hash (title, publishedAt); the bad values survive every refresh forever | 2026-07-10 (audit R2, #440): cap-0066's garbage title (a mid-document heading the H1 regex grabbed) and 100%-null dev-docs publishedAt would have outlived their extraction fixes — the body hash never changes, so the upsert marked every row `unchanged` | metadata-only drift path in `upsertChunks` (content-identical + title/publishedAt drifted → update in place, NO re-embed; dates compared by calendar day so ISO-vs-date formats don't churn the corpus daily); design rule: when shipping an extraction fix, ask "is the fixed field inside the skip-hash?" — if not, it needs its own drift check |
| 26 | **Source classification by crawl-origin** — records inherit the authority/freshness of the CRAWL that found them, not what they ARE; junk and low-tier pages ride a canonical source's trust | 2026-07-10 (audit R2, #439): meeting recaps + author-pagination mirrors crawled from developers.stellar.org ingested as `dev-docs` → 0.95 authority + evergreen freshness; a bare-date one-liner outranked CAP-0051, one recap filled 3 result slots via mirror URLs | rank-time reclassification (`meeting-notes` authority 0.5, URL-dated freshness), junk-URL drop + content-mirror collapse in `research-rank.ts`, corpus prune (gated `prune_execute` dispatch); design rule: classify by what the page IS (URL/shape), never by which crawler fetched it |
| 27 | **Post-filter after a capped pool** — a metadata filter applied AFTER bounded retrieval silently false-negatives: the pool never contained the matching rows, so "filtered to 0" reads as "doesn't exist" | 2026-07-19 (cold audit of the new surfaces): /api/research auditor/protocol/severity post-filtered the unscoped vector pool — a query whose top-K lacked audit chunks returned 0 even when perfectly-matching audit chunks existed ("this firm never audited this") | filters imply their scope AT retrieval (audit-metadata filters force source=audit, #596); contradictory explicit scope → 400; design rule: any filter on a bounded pool must narrow the FETCH, not just the page |
| 28 | **Single-affix extraction assumptions** — cleanup code assuming one brand-affix ordering ("Title \| Brand") destroys the mirrored ordering ("Brand \| Title"), and the damage hides in plain sight as a plausible value | 2026-07-19: stellar.org foundation-news og:titles are brand-PREFIX ordered; the suffix-only strip left ~50 posts titled just "Stellar" — the Protocol 27 "Zipper" upgrade guide was invisible to retrieval while a mainnet vote happened | handle both orderings (#593); golden eval counts BAD-TITLE docs; design rule: test extraction against every template family the source serves, not the first sample |
| 29 | **Hardcoded-count assertions rot into pipeline blockers** — an exact-N check on a growing set fails exactly when the system improves, and if it gates a pipeline the improvement is what gets blocked | 2026-07-19: the publish smoke test asserted "tools/list returns 15 tools"; the server (correctly) reached 20 and the stale assertion blocked BOTH pending npm releases (1.1.11, 1.1.12) | derive expected counts from the source of truth (registerTool count, #599) + invariants (name uniqueness) instead of literals; design rule: a count literal in a test is a rot timer |
| 30 | **New-endpoint checklist omissions recur in pairs** — the step that isn't mechanically enforced gets skipped by EVERY author, human or agent, in the same week | 2026-07-19: /api/audits shipped without its next.config publicApi entry (no CORS/X-API-Version); hours later the improvement-loop agent's /api/people missed the exact same entry | the extended drift check flags missing headers per endpoint (#594 caught both); design rule: when review catches a checklist miss, ask "what guard makes this class impossible" the same day — the second instance is already in flight |
| 32 | **Unverifiable read as absent** — a detector treats "I couldn't see it" as "it isn't there", so an upstream outage becomes our finding | 2026-07-23: self-audit called `api-client@1.5.0`/`@1.5.1` uninstallable on a transient npm **504** (both served 200 seconds later, and `1.4.0` passed in the same loop); the same run found report-liveness counting 5xx, timeouts, `EAI_AGAIN` and invalid TLS certs as positive evidence a project was **dead**, on the report that feeds demotion decisions | `src/lib/probe-external.ts` — one classifier for every third-party probe: **only** 404/410 or a host that doesn't resolve/refuses is `absent` (a finding); 5xx/timeout/bot-wall/bad-cert is `unverifiable`, retried once and reported in its own non-gating column. Unit-pinned in `probe-external.test.ts`. The third verdict only works if it ESCALATES — otherwise a permanently sick origin rests at "unverifiable" forever and leaves the dashboard: `check-links` therefore keeps two independent streaks (`consecutiveFailures` for proven-broken, `consecutiveUnverifiable` for no-verdict) and flips `needsReview` after 3 consecutive unverifiable runs (`link-history.ts`, unit-pinned). Design rule: a detector needs three verdicts, not two — and the third needs a clock, or it becomes a place findings go to die |

## The loop

1. An incident lands (sls item, tracker issue, demo feedback, our own audit).
2. Fix it **and** identify the class. New class → new row + guard; existing class → ask why the guard missed it and strengthen it.
3. Close the incident with reproducible probes; the guard keeps it fixed.
4. **Project the class forward**: ask "which OTHER surfaces have this same shape and haven't broken yet?" — file each projection as an entry in [/ideas/](../../ideas/) tagged `Projects from: class N`. This is the self-improving half of the loop: mistakes generate the improvement backlog before the next incident does.
5. Before shipping anything new, walk the table — the checklist is the "ahead of the curve" part.
